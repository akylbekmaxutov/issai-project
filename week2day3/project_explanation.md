# Настройка `.env` и объяснение проектов

## Часть 1. Как создать файл `.env` с токеном `OPENAI_TOKEN`

`.env` — это обычный текстовый файл, в котором хранятся секреты (например, API-ключ).
Он лежит в корне проекта и **не попадает в Git**, поэтому ключ остаётся приватным.

### Шаг 1. Создать файл `.env` в VS Code

1. Откройте папку проекта в VS Code: **File → Open Folder…** и выберите папку,
   где лежат `.py` файлы.
2. В панели **Explorer** (слева) нажмите на название папки правой кнопкой
   и выберите **New File** (или кнопку «новый файл» вверху панели).
3. Введите имя файла именно так — с точкой в начале:

   ```
   .env
   ```

> 📌 Файл должен лежать **в корне проекта**, рядом с `.py` файлами.

### Шаг 2. Записать токен внутрь

Откройте `.env` в VS Code и впишите одну строку:

```
OPENAI_TOKEN=sk-ваш_секретный_ключ_здесь
```

Сохраните файл: **Ctrl + S** (на Mac — **Cmd + S**).

Правила:
- **без пробелов** вокруг знака `=`
- **без кавычек** вокруг значения
- название переменной (`OPENAI_TOKEN`) должно точно совпадать с тем, что в коде:
  `os.getenv("OPENAI_TOKEN")`

### Шаг 3. Установить библиотеку для чтения `.env`

```bash
uv pip install python-dotenv openai pydantic
```

### Шаг 4. Как это работает в коде

В каждом проекте есть две строки:

```python
load_dotenv()                                   # читает файл .env
client = OpenAI(api_key=os.getenv("OPENAI_TOKEN"))  # берёт ключ из .env
```

`load_dotenv()` загружает переменные из `.env` в окружение, а `os.getenv("OPENAI_TOKEN")`
достаёт значение по имени. Если файл `.env` отсутствует или имя написано неверно —
ключ будет `None`, и появится ошибка авторизации.

## Часть 2. Как добавить `.env` в `.gitignore`

`.gitignore` — это файл со списком того, что Git **должен игнорировать**.
Добавив туда `.env`, вы гарантируете, что секретный ключ никогда не уйдёт в репозиторий.

### Шаг 1. Создать файл `.gitignore` в VS Code (если его ещё нет)

В панели **Explorer** правой кнопкой по папке проекта → **New File** →
введите имя:

```
.gitignore
```

### Шаг 2. Добавить `.env` в список

Откройте `.gitignore` в VS Code и впишите строку:

```
.env
```

Сохраните файл: **Ctrl + S** (на Mac — **Cmd + S**).

### Шаг 3. Проверить, что Git игнорирует файл

```bash
git status
```

Файла `.env` **не должно быть** в списке изменений. Если он там есть —
значит, он уже был добавлен в Git раньше. Удалите его из отслеживания (файл на диске останется):

```bash
git rm --cached .env
```

После этого `.env` снова будет игнорироваться.

> ⚠️ Если ключ уже был случайно отправлен в Git — считайте его скомпрометированным
> и выпустите новый токен в кабинете OpenAI.

## Часть 3. Объяснение кода всех проектов

Все пять проектов построены на **одной идее**: модель возвращает не просто текст,
а **структурированный объект** (Pydantic-модель). Это позволяет сразу использовать
ответ в коде — без разбора строк.

### Общая часть (есть во всех файлах)

```python
load_dotenv()
client = OpenAI(api_key=os.getenv("OPENAI_TOKEN"))
MODEL = "gpt-4o-mini"
```

- `load_dotenv()` — загружает токен из `.env`.
- `client` — объект для общения с OpenAI.
- `MODEL` — какая модель используется.

Ключевая команда во всех проектах:

```python
completion = client.chat.completions.parse(
    model=MODEL,
    messages=[...],
    response_format=SomeModel,   # <- модель Pydantic = форма ответа
)
result = completion.choices[0].message.parsed   # уже готовый объект, а не текст
```

`response_format` заставляет модель отвечать строго в заданной структуре.
`.parsed` возвращает готовый Python-объект с проверенными полями.

### Проект 1 — `project_1_classifier.py` (Классификатор отзывов)

**Что делает:** превращает свободный текст отзыва в структурированные метки.

Модель `Classification` описывает результат:
- `sentiment` — тональность: `positive / negative / neutral / mixed`;
- `confidence` — уверенность от 0 до 1;
- `topics` — 2–4 коротких тега темы;
- `urgent` — нужен ли срочный ответ (`True/False`).

Функция `classify(feedback)` отправляет отзыв модели и получает объект `Classification`.
В конце программа печатает результат, добавляя 🚨, если отзыв срочный.

**Главная идея:** свободный текст → готовые поля для дашборда или базы данных.

---

### Проект 2 — `project_2_extractor.py` (Извлечение информации)

**Что делает:** достаёт из «человеческого» сообщения чистую структурированную запись.
Это самый частый реальный сценарий использования структурированного вывода.

Модель `Meeting` описывает встречу:
- `title` — название;
- `date`, `time` — дата и время (могут отсутствовать → `None`);
- `participants` — список участников;
- `location` — место;
- `action_items` — список задач «после встречи».

Функция `extract(message)` принимает обычное сообщение (например, «давайте созвонимся
во вторник в 15:00…») и возвращает заполненный объект `Meeting`.

**Главная идея:** беспорядочный текст → аккуратная запись с полями.

---

### Проект 3 — `project_3_quiz.py` (Генератор тестов)

**Что делает:** создаёт готовый тест с вариантами ответов.

Две модели:
- `Question` — один вопрос: текст, ровно 4 варианта `options`,
  индекс правильного ответа `correct_index`, объяснение `explanation`;
- `Quiz` — тема `topic` и список вопросов `questions`.

Функции:
- `make_quiz(topic, n, difficulty)` — генерирует тест из `n` вопросов;
- `run_quiz(quiz)` — запускает интерактивный тест в терминале:
  печатает вопросы, читает ввод через `input()`, считает баллы и показывает счёт.

**Главная идея:** структурированный тест можно сразу подать в интерфейс, базу или проверяльщик —
без разбора строк.

---

### Проект 4 — `project_4_planner.py` (Планировщик целей)

**Что делает:** из расплывчатой цели делает упорядоченный пошаговый план.
Это первое знакомство с навыком агента — **ДЕКОМПОЗИЦИЯ ЗАДАЧИ**
(разбиение большой цели на шаги).

Две модели:
- `Task` — один шаг: номер `step`, название `title`,
  приоритет `priority` (`low/medium/high`), оценка времени `estimated_minutes`;
- `Plan` — цель `goal`, краткое описание `summary` и список задач `tasks`.

Функция `make_plan(goal)` принимает цель и возвращает готовый план.
В конце программа печатает шаги и суммирует общее время.

**Главная идея:** большая цель → конкретные шаги с приоритетами и временем.

---

### Проект 5 — `project_5_router.py` (Мини-агент-маршрутизатор)

**Самый важный проект.** Здесь модель уже не просто создаёт текст —
она **РЕШАЕТ, что делать**. Это зерно настоящего агента: паттерн «реши → действуй»
(decide-then-act).

Три части:

1. **Структурированное решение** — модель `Decision`:
   - `action` — какое действие выбрать: `calculate / reverse_text / word_count / chitchat`;
   - `argument` — вход для выбранного действия;
   - `reasoning` — одно предложение, почему выбрано это действие.

2. **«Инструменты» — обычные Python-функции:**
   - `calculate(expr)` — вычисляет выражение (через `eval` — только для учебного демо!);
   - `reverse_text(text)` — переворачивает строку;
   - `word_count(text)` — считает слова;
   - `chitchat(text)` — обычный ответ модели.

   Все они собраны в словарь `TOOLS`.

3. **Цикл «реши, потом действуй»** — функция `route(user_input)`:
   - модель возвращает объект `Decision`;
   - ваш код запускает нужную функцию: `TOOLS[decision.action](decision.argument)`;
   - печатается результат.

**Главная идея:** модель выбирает действие, а **ваш Python-код** его выполняет.
Завтра (День 4) эти «действия» станут настоящими инструментами агента.

## Краткая шпаргалка

```bash
# 1. установить зависимости
uv pip install python-dotenv openai pydantic

# 2. запустить любой проект
uv run python project_1_classifier.py
```

Файлы `.env` и `.gitignore` создаются в **VS Code** (правой кнопкой по папке → **New File**):

```
.env  →  OPENAI_TOKEN=sk-ваш_ключ
.gitignore  →  .env
```
